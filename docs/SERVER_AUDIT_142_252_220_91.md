- There is no built-in health endpoint or metrics endpoint.

## Stability Gaps

1. Production server source for this exact binary is not present in the current iOS fork.
2. Service has restart policy and high fd limit, but no readiness check.
3. Health must currently be inferred from process state, UDP socket, file logs and WireGuard handshakes.
4. Logs are plaintext append-only file with no structured fields and no logrotate unit visible from this audit.
5. No metrics:
   - active sessions;
   - session accept/close counts;
   - handshake failures;
   - backend read/write errors;
   - bytes in/out;
   - last activity.
6. No dry-run deployment path documented for a second port.

## Server Implementation Direction

Build a fork-owned server source in this repository before touching production:

```text
cmd/vk-turn-proxy-server
internal/serverhealth or internal metrics package later
```

Status: initial `cmd/vk-turn-proxy-server` now exists locally and supports the current `-srtp` production mode plus admin health/metrics endpoints.

Initial compatibility target:

```bash

  -listen 0.0.0.0:56004 \
  -connect 127.0.0.1:51820 \
```

Add new flags in backward-compatible form:

```text

```

Safe rollout:

1. Build new binary locally and test with loopback UDP backend. Done locally.
2. Upload as `/tmp/vk-turn-proxy-server.next`. Done for short-lived dry-run.
3. Run dry-run process on localhost-only second port `127.0.0.1:56014` with admin `127.0.0.1:56085`. Done with `timeout`; production untouched.
4. Verify `/healthz`, `/readyz`, `/metrics`. Done.
5. Test one client against the second port.
6. Backup current binary and unit.
7. Switch production only after successful second-port test.
8. Roll back by restoring the old binary path or unit `ExecStart`.
